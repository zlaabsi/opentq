# March-April 2026 notes

This file records the sources that drove the initial architecture.

## Weight formats

- `turbo-tan/llama.cpp-tq3` publicly documents the runtime side for `TQ3_1S` and `TQ3_4S`, and explicitly states that the quantization tooling is not part of the public fork.
- `YTan2000/Qwen3.5-27B-TQ3_4S` documents `TQ3_4S` as a 3.5-bit Walsh-Hadamard-transform weight format with four per-8 scales per 32-weight block.
- `cksac/turboquant-model` shows that an open TurboQuant-style weight path is viable, including residual `4+2` and `4+4` variants.

## Decode-time acceleration

- `DFlash: Block Diffusion for Flash Speculative Decoding` shows that speculative drafting no longer needs to be autoregressive. The key architectural implication is that the custom runtime should keep a clean hook for hidden-state extraction and parallel verify.
- `vllm-project/speculators` matters because it turned DFlash into something trainable and deployable rather than just a paper idea.
- `dflash-mlx` matters because it demonstrates that exact speculative decoding on Apple Silicon is already practical enough to justify a dedicated runtime path, even though MLX itself does not ship speculative primitives.

## Apple Silicon-specific inference

- `Native LLM and MLLM Inference at Scale on Apple Silicon` is a reminder that Apple-native stacks can win when concurrency and prefix caching matter.
- `Open-TQ-Metal` is the strongest signal that compressed-domain attention on Apple Silicon should be treated as a separate runtime track instead of an afterthought.
- Ollama's March 30, 2026 MLX preview confirms that the Apple stack is moving fast on quantized production-style runtimes.

## llama.cpp-specific signals

- `llama.cpp` now contains Hadamard generation inside the KV-cache path for quantized attention rotation.
- The April 2026 line also introduced fresh low-bit backend work such as `Q1_0`, which reinforces the idea that custom low-bit tensor types are reasonable to target upstream.
- Community experimentation around entropy-adaptive KV allocation suggests that per-head and per-role policies are becoming practical, especially for hybrid models.

## Decision taken here

- build the weight family first in public
- keep the `llama.cpp` path concrete via GGUF export + patchsets
- reserve the more opinionated runtime ideas for `opentq-metal`
